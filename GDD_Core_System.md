## 1. 武器升級系統

### 1.1 設計原則
- **不採用稀有度分級**(無白/藍/紫/金裝之分),所有同名武器透過強化數字區分強弱
- 武器強度完全反映在「+[數字]」上,例如「唐刀 +7」

### 1.2 核心機制
| 項目 | 說明 |
|---|---|
| 升級方式 | 消耗「強化石」+ 金幣,每次升級有成功率 |
| 等級表示 | 武器名稱後綴 +0 ~ +N(如「開山刀 +12」) |
| 失敗懲罰 | 高階強化失敗可能降級(是否損毀待定) |
| 保底機制 | 連續失敗N次後,下次必定成功 |

### 1.3 資源來源
- 強化石:練功掉落、地盤守衛練功掉落、社團商店兌換
- 金幣:練功、地盤產出

### 1.4 數值設計注意
- 每級屬性加成建議採遞增曲線,避免後期數值爆炸
- 升級結果為權威伺服器運算(Colyseus Room內處理),防止用戶端竄改成功率

### 1.5 待補充
- [ ] 高數字強化失敗是否會損毀武器,或僅降級?
- [ ] 是否有數字上限(如+20封頂)?

## 2. 社團系統(原公會系統)

### 2.1 設計目標
- 建立玩家社群黏著度與歸屬感
- 作為地盤爭奪的組織單位
- 職級與職能參考香港三合會架構,呼應《古惑仔》世界觀

### 2.2 職級架構(參考三合會職銜,依實際遊戲需求命名)

| 職級 | 對應現實稱謂 | 權限範圍 |
|---|---|---|
| 話事人(社團長) | 龍頭 | 所有權限,唯一可解散社團、任免所有職級 |
| 坐館 | 坐館 | 副手,權限接近話事人,可任免中層職級 |
| 紅棍 | 紅棍(打手頭領) | 負責地盤佔領戰調度、守衛部署權限 |
| 白紙扇 | 白紙扇(軍師) | 負責社團倉庫/金錢管理,無人事任免權 |
| 草鞋 | 草鞋(聯絡) | 負責招募新成員、處理入會申請 |
| 四九仔 | 四九(基層成員) | 一般成員,僅能使用基礎功能(倉庫存取、商店購買) |

> 核心設計重點:**權限依職能分離**,例如「白紙扇」只能動用金錢/倉庫物資,不能升遷或開除成員;「草鞋」只能招募,不能碰錢。避免單一職位權力過度集中,同時貼合三合會分工文化。

### 2.3 核心機制
| 項目 | 說明 |
|---|---|
| 建立條件 | 玩家等級門檻 + 金幣費用 |
| 成員上限 | 依社團等級提升(貢獻度累積經驗值升級) |
| 社團倉庫 | 依職級分級存取權限(如上表) |
| 貢獻度 | 成員捐獻金幣/材料換取貢獻度,可於社團商店兌換物品 |

### 2.4 社團商店
- 獨立於一般NPC商店,只能用「貢獻度」購買
- 販售社團專屬物品/裝備(一般商店買不到),作為社團向心力的誘因之一
- 具體商品清單待定(建議與武器強化石、稀有材料掛鉤)

### 2.5 技術考量
- 社團資料獨立於GameRoom外,用專屬資料庫/Room管理,避免與戰鬥Room耦合
- 職級權限判斷建議做成獨立的權限矩陣(permission matrix),方便日後調整職級或新增職能而不用改動邏輯結構

### 2.6 明確排除範圍
- **暫不開發「社團戰」系統**(正式的社團 vs 社團宣戰/排程對戰機制),理由:系統複雜度高,且參考《古惑仔Online》原作做法,社團之間的衝突是**由玩家自行發起**(例如相約佔地盤時單挑或打群架),而非由系統提供正式宣戰功能

### 2.7 待補充
- [ ] 職級任免的具體操作介面(誰能一次任免幾人)?
- [ ] 貢獻度捐獻的比例與上限?

## 3. 佔領地盤系統

### 3.1 設計目標
- 提供社團層級的長期經營目標與資源產地
- 降低伺服器負擔與開發複雜度,採**隨時可佔領**而非排程開戰制

### 3.2 核心機制
| 項目 | 說明 |
|---|---|
| 佔領時機 | **無固定開放時段**,任何時間只要地盤守備被擊破即可佔領 |
| 佔領方式 | 攻方擊敗地盤內的守衛(NPC化的敵方社團崗位)即可換旗 |
| 持有收益 | 持有社團可招聘部署守衛(0~10個,依投入資源決定),守衛練功產出部分歸公 |
| 社團升級 | 佔領地盤本身也計入社團貢獻度,協助社團升級 |

> 採「隨時可佔領」而非排程戰的原因:排程戰需要額外的戰鬥Room排程、時段管理、通知系統,對單週開發時程負擔過大;隨時佔領則只需在守衛被清空時觸發一次歸屬權轉移判定,邏輯單純很多。

### 3.3 守衛系統(核心特色)

| 項目 | 說明 |
|---|---|
| 數量上限 | 每塊地盤**固定最多10個守衛位**(全域固定值,非每塊地盤可調參數) |
| 招聘制 | 守衛**不是佔地後自動滿額**,而是每個守衛位都要社團**個別招聘(付費)** 才會有守衛進駐,因此實際守衛數量可以是0~10之間任意數量,由社團資源投入決定 |
| 設定權限 | 依社團職級權限開放(建議「紅棍」以上可招聘/部署) |
| 守衛功能① | 開放社團成員在其範圍內練功(打守衛刷經驗),等級越高的守衛給的經驗值越多(參考《萬王之王Online》的怪物崗哨概念) |
| 守衛功能② | 守衛同時是防禦手段,守衛強度越高/數量越多,地盤越不容易被其他社團攻破 |

- 守衛可視為「社團自建的練功怪」,社團等級/資源投入越多,能招聘的守衛等級越高、招滿10個位的可能性也越高
- 守衛練功產出(經驗、掉落)建議與野外練功有差異化設計(例如稀有材料掉落率較高),讓玩家有動機主動加入有經營地盤的社團
- 招聘制帶來的設計效果:剛佔領的地盤守衛數量可能是0,防禦力薄弱,給予其他社團「趁虛而入」的空間;社團需要持續投入資源才能把守衛補滿10個,形成長期的資源投入誘因

### 3.4 技術考量
- 佔領事件觸發後的歸屬權轉移建議用Colyseus的Room lifecycle管理該地盤當下的戰鬥狀態
- 地盤持有狀態與守衛配置必須落地儲存(資料庫持久化),不可只存在記憶體State,避免伺服器重啟後地盤歸屬遺失

### 3.4a Dashboard — Map Management:地盤設置模組(新增)

#### 基本定位
在現有Dashboard的Map Management工具中新增一個「地盤設置」模組,供開發者/GM在地圖上直接繪製、標記地盤範圍。

#### 地盤形狀與繪製方式
| 項目 | 說明 |
|---|---|
| 領地形態 | **完全貼合地面的平面區域**(2D多邊形貼在地表,非3D體積),渲染上可用半透明色塊表示範圍 |
| 建立流程 | 進入「新增領地」模式後,**每次click在地圖上落一個點,依序連成一條線**;最後一點click回到起始點時,系統自動判定封閉並完成該領地的範圍設定 |
| 選擇此做法的原因 | 逐點連線是最直覺的操作方式,不需要額外UI元件(如拖曳控制點、輸入座標框),GM上手成本最低,也最貼近「畫一塊地」的直覺 |

#### 領地列表與刪除
| 項目 | 說明 |
|---|---|
| 列表面板 | 另開一個列表區域,列出所有已建立的領地(名稱/ID) |
| 用途 | 當地圖上點位太密集、難以直接點選到某塊領地時,可從列表選取 |
| 刪除方式 | 於列表中選取(highlight)特定領地後,提供刪除按鈕移除該領地 |

#### 修改功能(僅限移動既有頂點)
| 項目 | 說明 |
|---|---|
| 允許操作 | 拖曳既有頂點調整位置,盡量利用空間貼合地形 |
| 禁止操作 | **不可新增或刪除頂點**,頂點數量固定在建立當下所畫的數量 |
| 回溯機制 | 進入編輯模式時記錄「讀取當下」的版本快照,若對修改結果不滿意,可一鍵回溯到進入編輯前的版本 |
| 自我相交防呆 | 拖曳頂點若導致**任兩個頂點重疊**,或**任兩條邊線交叉**(使多邊形無法構成有效封閉面),則**直接擋掉該次拖曳**,頂點回彈到拖曳前的位置,不允許放開成立 |

> 設計原因:允許移動頂點但鎖定數量與新增/刪除,是在「完全不能改」與「自由編輯」之間取一個折衷 —— GM能微調範圍貼合地形,但不會讓程式邏輯需要處理動態增減頂點的複雜情況(如重新三角化、重新判斷封閉性)。回溯機制則降低GM亂調的心理負擔。自我相交防呆同理:與其允許扭曲形狀後再處理渲染/判定錯誤,不如在互動當下就擋掉,邏輯最單純。

> 建立新領地時的逐點連線流程也適用同一條規則:落下新點時,若該點與既有點重疊,或新畫出的邊與既有邊交叉,同樣直接擋掉該次click,不落點。

#### 輔助顯示功能
- 提供「顯示/隱藏所有大廈(建築物模型)」的開關,方便工作人員在畫領地時用肉眼確認地形與街道邊界,不被建築物模型擋住視線

#### 鏡頭/視角操作
| 操作 | 對應行為 |
|---|---|
| Space + 滑鼠左鍵拖曳 | 平移鏡頭(上下左右移動) |
| 滑鼠右鍵拖曳 | 旋轉視角(轉角度) |
| 滑鼠滾輪 | 縮放(zoom in/out) |

> 注意:操作模式需與「新增領地」的click落點模式互不衝突 —— 一般click(無Space)落點畫線,Space+左鍵拖曳才是平移,避免誤觸多畫出不必要的點。

#### 資料輸出與銜接
- 產出的地盤定義(封閉多邊形頂點座標陣列、ID、名稱)寫入設定檔或資料庫,供伺服器端佔領/守衛系統讀取
- 地盤範圍資料應與3.4節的持久化地盤狀態資料表對應(同一份地盤ID),避免Dashboard定義的地盤與伺服器實際判定的地盤範圍不一致
- 建議欄位至少包含:地盤ID、名稱、多邊形頂點座標陣列、守衛上限(固定10,對應3.3節,非可調欄位)
- **Dashboard不設定「擁有者」欄位**:所有透過此模組建立的地盤,初始狀態一律為**中立/無主**,擁有權完全交由3.2節的佔領機制(擊敗守衛換旗)在遊戲運行時動態決定,不由GM在編輯地盤範圍時手動指定,避免靜態編輯資料與伺服器實際戰鬥結果的歸屬狀態互相衝突
- **頂點數量不設上限**,保留GM畫出圓形或不規則造型領地的彈性(用多個短邊模擬曲線)
- **領地範圍不可重疊**:新增或修改(移動頂點)時,系統需即時檢查多邊形是否與其他既有領地範圍相交,若相交則阻擋該操作並提示衝突

#### 權限
- 建議僅限開發者/GM帳號可進入此模組,一般玩家不可見

#### 待補充
- [ ] 多邊形相交檢查的實作方式(如採用標準的polygon intersection演算法)

### 3.5 循環設計
- 佔地 → 部署守衛 → 社團成員在守衛處練功產出資源 → 資源用於武器強化 → 更有戰力守住/擴張地盤

### 3.6 待補充
- [ ] 守衛的等級/強度上限與部署所需資源

## 4. 練功(狩獵/升等)系統

### 4.1 設計目標
- 作為玩家獲取經驗值、金幣、材料的主要日常迴圈
- 承接你現有的PvE戰鬥與敵人AI系統
- 與地盤守衛系統形成差異化練功選擇(野外 vs 地盤守衛)

### 4.2 核心機制
| 項目 | 說明 |
|---|---|
| 練功區域 | 依等級分區,沿用現有Enemy管理系統 |
| 地盤守衛練功 | 社團成員可在己方(或已加入之)社團地盤內打守衛練功,守衛等級越高經驗越多 |
| 經驗曲線 | 建議採平滑遞增公式,避免高等級卡關過久 |
| 掉落系統 | 一般材料(高機率)、強化石(中低機率)、金幣 |

### 4.3 與其他系統的連結
- 練功/守衛掉落 → 供應武器升級所需的強化石/材料
- 練功獲得金幣 → 可用於社團捐獻換取貢獻度、部署守衛

### 4.4 技術考量
- 已有實作基礎(GameRoom + Schema同步的Enemy系統),主要待補的是**數值表**(經驗曲線、掉落率)
- 建議將掉落表與數值曲線抽成設定檔(JSON/CSV)
- 守衛練功邏輯可重用一般敵人的戰鬥/掉落框架,差異只在於「守衛」的歸屬社團標記與經驗加成係數

### 4.5 待補充
- [ ] 野外練功與守衛練功的經驗/掉落差異倍率
- [ ] 非社團成員能否使用其他社團的守衛練功(可能需守衛開放設定)?

## 5. 組隊系統

### 5.1 設計目標
- 鼓勵玩家共同練功、共同社交

### 5.2 核心機制
| 項目 | 說明 |
|---|---|
| 組隊人數 | 待定(建議先抓一個小隊規模,如4~6人) |
| 經驗分配 | **同一份經驗值由全隊分享**(而非各自獨立計算),擊殺一隻怪的經驗值依隊伍人數分配給所有在場隊員 |
| 分配方式 | 待定:平均分配 / 依貢獻度(傷害量)分配 |

### 5.3 技術考量
- 組隊狀態建議獨立於GameRoom的Party概念管理,擊殺結算時查詢隊伍成員清單並分配經驗
- 需考慮隊員分散在不同Room(不同練功區)時經驗是否仍分享,或僅限同Room內

### 5.4 待補充
- [ ] 組隊人數上限
- [ ] 經驗分配公式(均分 or 依貢獻)

## 6. 系統間關聯圖(文字版)

```
練功系統 ──(掉落資源)──► 武器升級系統(+數字強化,無稀有度)
   │  ▲                        ▲
   │  │(地盤守衛練功)          │
   ▼  │                        │
金幣/材料 ──► 社團系統 ──► 佔領地盤系統(隨時佔領+守衛)
                │                    │
                ▼                    │
           社團商店(貢獻度兌換) ◄────┘(佔地計入社團貢獻)

組隊系統 ──(經驗共享)──► 疊加於練功系統之上,不獨立產出資源
```
