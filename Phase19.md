Gangs Online: 任務藍圖系統技術說明書 (Phase 0.16.4)1. 系統概述 (System Overview)本系統為《Gangs Online》開發的核心組件，旨在建立一個可視化的節點式任務編輯器。透過圖形化介面定義任務的生命週期，並將邏輯輸出為標準 JSON 數據，供伺服器驗證與客戶端演出。設計核心原則：原子化設計 (Atomic Design)：每個任務 ID 為獨立文件，降低系統耦合度，方便 Debug 與未來擴充。手動接取節奏 (Manual Pacing)：遵循經典 MMORPG 體驗，連環任務之間不自動彈出，需玩家主動與 NPC 交互，提供劇情緩衝感。性能優化：利用客戶端實例化 (Phasing) 技術處理任務中途的 NPC，減輕伺服器負擔並提升玩家沉浸感。2. 三端職責與協作架構模組運作環境核心職責管理儀表板 (Dashboard)Web (React Flow)提供設計師編輯畫布、配置節點屬性、執行邏輯驗證（如死循環檢查）。邏輯執行引擎 (Server)Game Server解析 JSON、驗證玩家資格（等級/道具）、紀錄任務進度旗標、發放獎勵。遊戲呈現層 (Client)Game Client渲染任務 UI、處理對話演出、動態渲染任務 NPC (Phasing)、提供路徑追蹤導航。3. 節點定義與屬性配置 (Node Specifications)編輯器需實作以下自定義節點，每個節點需具備側邊欄編輯面板供設計師填寫參數：3.1 流程與對話節點開始節點 (Start Node)：觸發點：必須選擇並掛載於特定的 NPC 模板 ID。前置檢查：設定最低/最高等級限制、指定任務 ID 狀態（需檢測特定任務是否為「已完成」）。對話節點 (Dialogue Node)：演出屬性：說話者 ID、表情代碼、多語言文本。支援設置多段連續對話序列。選擇節點 (Choice Node)：分支邏輯：設置多個選項按鈕文字，分別連線至不同的後續邏輯節點。結束節點 (End Node)：結算行為：定義獎勵清單（經驗值、HKD、道具 ID 與數量）。執行後更新伺服器端的玩家任務旗標。3.2 目標與判定節點任務目標 (Task Node)：收集目標：指定道具 ID 及其所需數量（需檢測背包）。擊殺目標：指定敵人模板 ID 及其所需數量。交互目標：要求玩家與下一個特定的 NPC 模板進行對話。位置目標：要求玩家在限時內抵達指定的座標座標區域。條件判定 (Condition Node)：即時檢查：檢測金錢餘額、持有指定道具數、全域變數狀態。支援「成功」與「失敗」兩條輸出路徑。動作執行 (Action Node)：系統指令：扣除任務所需道具/金錢、觸發特定客戶端實體 (Client-side NPC) 的渲染狀態。4. 關鍵業務邏輯 (Core Business Rules)4.1 連環任務實作 (Quest Chaining)解耦邏輯：任務 A 與任務 B 在物理檔案上完全獨立。任務 B 的「開始節點」僅會檢查玩家資料庫中的 Completed_Quests 列表中是否存在任務 A 的 ID。交互儀式：任務 A 完成後，玩家必須再次與 NPC 主動對話才能觸發任務 B 的啟動流程，確保遊戲節奏具備「喘息感」。4.2 客戶端實體化機制 (Phasing)動態 NPC：任務流程中涉及的「非起始 NPC」（即中途出現的角色）應僅在客戶端進行渲染，減少伺服器管理實體的負擔。數據回傳：雖然 NPC 在客戶端顯示，但涉及對話完成、道具獲取等核心數值變動，必須回傳伺服器進行最終校驗以防作弊。4.3 任務狀態機 (State Machine)系統需管理每位玩家對應各任務 ID 的四種狀態：Locked (鎖定)：未滿足前置任務要求或等級不足。Available (可接取)：滿足所有要求，NPC 上方顯示驚嘆號等視覺提示。Active (進行中)：任務已接取，任務日誌顯示當前目標。Completed (已完成)：獎勵已發放，紀錄存入歷史列表，無法重複接取（除非為日常任務）。5. Dashboard 編輯器功能要求 (React Flow)視覺化驗證：編輯器需能自動偵測「斷頭路」（未連接至 End 的路徑）與「邏輯死循環」。物件選擇器：整合 NPC 模板庫與道具庫，設計師應能直接檢索名稱並選取對應 ID。JSON 導出：將 UI 位置數據（座標）與邏輯數據分離，確保後端僅讀取輕量化的邏輯圖表。6. 異常與邊界情況處理放棄任務：玩家放棄任務後，重置該 ID 狀態為 Available，並自動清除該任務在客戶端產生的所有臨時 NPC。空間檢查：在發放獎勵前需檢查玩家背包剩餘格子。若空間不足，應暫停結算並提示玩家清理背包。
